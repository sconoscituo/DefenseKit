# 기술 스택 설명

## C++17 선택 이유

방산/임베디드 환경에서 C++17이 선호되는 이유:

1. **결정론적 실행 시간**: 가비지 컬렉션 없음, 예측 가능한 메모리 관리
2. **하드웨어 근접 제어**: 직접 메모리 주소 접근, DMA 제어
3. **성능**: Zero-cost abstraction, 인라인 최적화
4. **구조적 바인딩 (C++17)**: `auto [x, y] = get_position();`
5. **`if constexpr`**: 컴파일 타임 분기로 템플릿 최적화
6. **표준 라이브러리**: `std::array`, `std::optional`, `std::variant`

## 수치 알고리즘 선택 근거

### Runge-Kutta 4차 vs 오일러법

| 항목 | 오일러 (1차) | RK4 (4차) |
|------|------------|-----------|
| 정확도 | O(h) | O(h⁴) |
| 계산 비용 | 1 함수 평가/스텝 | 4 함수 평가/스텝 |
| 안정성 | 낮음 | 높음 |
| 방산 적용 | 부적합 | 표준 |

탄도 계산에서 RK4를 사용하는 이유: 동일 스텝 크기에서 오차가 16배 감소.

### 칼만 필터 vs 확장 칼만 필터 (EKF)

이 프로젝트는 **선형 칼만 필터** 사용:
- 레이더 표적 추적: 직교 좌표 측정 → 선형 시스템
- EKF는 극좌표(거리/방위각) 측정 시 필요

실제 방산 시스템에서는 UKF (Unscented KF) 또는 IMM (Interacting Multiple Model) 사용.

### Cooley-Tukey FFT 반복적 구현

재귀 vs 반복:
- **재귀**: 코드 간결, 스택 오버플로우 위험
- **반복 (이 구현)**: 임베디드 적합, 스택 사용량 O(1)

비트 역순 치환 후 버터플라이 연산 순서:
```
len=2 → len=4 → len=8 → ... → len=N
```

## 행렬 라이브러리 직접 구현 이유

Eigen 대신 직접 구현한 이유:
1. **외부 의존성 최소화**: 방산 SW 인증(DO-178C) 시 써드파티 라이브러리 인증 부담
2. **크기 제한**: 칼만 필터는 4x4 이하 → 범용 라이브러리 오버킬
3. **이식성**: 임베디드 타겟 컴파일러 호환성
4. **이해도 증명**: 포트폴리오로서 알고리즘 직접 구현 능력 시연

## 실시간 시스템 고려사항

이 코드가 실제 방산 임베디드에 적용될 때 추가로 필요한 것:

```cpp
// 동적 할당 금지 → 정적/스택 메모리 사용
// 나쁜 예:
std::vector<TrajectoryPoint> traj = sim.simulate(...);

// 좋은 예 (임베디드):
static TrajectoryPoint traj_buffer[MAX_POINTS];
int traj_count = sim.simulate_static(traj_buffer, MAX_POINTS, ...);
```

**MISRA C++ 준수 포인트:**
- 예외 사용 최소화 (임베디드에서 비용)
- `goto` 금지
- 암묵적 형변환 금지
- 동적 메모리 할당 금지 (MISRA C++:2008 Rule 18-4-1)

## CMake 빌드 시스템

방산 프로젝트에서 CMake를 쓰는 이유:
- 크로스 컴파일 지원 (x86 → ARM Cortex-A/R/M)
- 타겟별 컴파일 플래그 관리
- CTest 통합으로 자동화 테스트
- CPack으로 배포 패키지 생성

크로스 컴파일 예시:
```bash
cmake .. -DCMAKE_TOOLCHAIN_FILE=arm-none-eabi.cmake \
         -DCMAKE_BUILD_TYPE=Release
```
